# 从LLM对话到Wiki知识库

将用户与LLM的对话内容转化为结构化的wiki知识。核心流程：获取对话 → 提取概念 → 评估掌握 → 纠正误解 → 推荐存储位置 → 准备素材 → 执行写作。

## 阶段 1: 获取对话内容

### 1.1 尝试直接获取

使用 webfetch 工具获取用户提供的URL内容。如果内容完整（能提取出有意义的对话结构），直接使用。

### 1.2 内容不完整时启用Browser

如果 webfetch 获取的内容明显不完整（缺少用户消息、缺少AI回复、内容截断），使用 agent-browser skill 让agent翻阅浏览器完整获取对话内容。

**判断标准：**
- 内容长度异常短
- 缺少对话双方的消息标记
- 只有部分内容，明显有截断
- 包含 "..." 或 "loading" 等占位符

### 1.3 提取对话结构

将获取的原始内容整理为清晰的对话格式：

```
[用户]: ...
[AI]: ...
[用户]: ...
[AI]: ...
```

## 阶段 2: 提取核心概念与知识点

### 2.1 分析对话内容

从对话中提取：
- **核心概念**：本次对话涉及的主要技术概念/算法/理论
- **关键问题**：用户提出的核心疑问
- **解答要点**：AI给出的关键解释
- **代码/公式**：对话中出现的任何代码片段或数学公式
- **类比/比喻**：AI用来解释复杂概念的类比
- **用户误解点**：对话中暴露的用户理解偏差（如果有）

### 2.2 输出提取摘要

向用户展示提取结果，格式如下：

```
📋 对话内容摘要

核心概念: [概念1], [概念2], ...
关键问题: [问题描述]
主要解答: [解答要点]
涉及代码/公式: [是/否，如有则列出]

─────────────────────
```

## 阶段 3: 评估掌握程度（关键步骤）

**这是本技能的核心价值所在。对话只展示了学习过程，不代表掌握。**

### 3.1 生成验证问题

根据提取的核心概念，生成3-5个关键验证问题。问题设计原则：
- **覆盖核心机制**：不仅问"是什么"，还要问"为什么"和"怎么用"
- **检测常见误区**：针对这个概念最容易误解的点提问
- **要求应用**：让用户用简单例子说明概念
- **关联比较**：与其他相似概念的区别

**示例：**
如果概念是 "Ray Tracing 中的 BVH"

❌ 差问题："什么是BVH？"（对话中已经问过）
✅ 好问题："BVH 构建时为什么通常使用 SAH 而不是简单的中点分割？"
✅ 好问题："如果场景中有大量动态物体，BVH 需要每帧重建吗？有没有优化方法？"
✅ 好问题："BVH 和 KD-Tree 在射线求交时的主要性能差异是什么？"

### 3.2 提问与评估

逐一向用户提问。对每个回答：
- **正确**：简要确认，继续下一个
- **部分正确**：指出遗漏点，补充说明
- **错误/不确定**：详细解释正确理解，用对话中的内容或新的类比帮助理解

**评估输出格式：**
```
❓ 问题 N: [问题]
你的回答: [用户回答]
评估: [正确/部分正确/需纠正]
[如有问题，给出纠正说明]
```

### 3.3 总结掌握状态

所有问题结束后，输出：

```
📊 掌握度评估

概念: [概念名]
掌握度: [扎实/基本理解/需要巩固/存在误区]
强项: [用户理解较好的部分]
待加强: [需要进一步理解的部分]
建议: [后续学习建议]

─────────────────────
```

## 阶段 4: 识别并纠正误解

### 4.1 对比对话内容与用户回答

在阶段3中，如果用户回答与对话中的解释存在偏差，或用户表现出新的误解：

1. **记录误解**：明确记录用户的错误理解
2. **分析原因**：为什么会这样理解？（概念相似性、过度推断、信息缺失）
3. **给出纠正**：用更清晰的方式重新解释，可以：
   - 引用对话中的正确解释
   - 使用新的类比
   - 用反例说明为什么原来的理解不对

### 4.2 输出纠正记录

```
🔧 误解纠正记录

误解: [用户的错误理解]
原因: [为什么会这样理解]
纠正: [正确理解]
```

## 阶段 5: 推荐Wiki存储位置

### 5.1 查阅现有Wiki结构

读取 `wiki/_master_index.md` 了解现有topic结构。
如果 `_master_index.md` 不存在或为空，则建议创建新topic。

### 5.2 匹配最佳Topic

根据提取的核心概念，推荐1-3个可能的存储位置：

```
📁 Wiki存储建议

选项1: [现有topic]/[新页面名]
  - 理由: [为什么适合]
  - 相关页面: [已有的相关wiki页面]

选项2: [现有topic]/[新页面名]
  - 理由: [为什么适合]

选项3（新建Topic）: [新topic名]/[新页面名]
  - 理由: [为什么现有topic都不合适]
```

### 5.3 与用户确认

询问用户偏好：
- "你觉得哪个选项最合适？"
- "或者你有其他想法？"
- 如果用户对推荐不满意，使用 brainstorming skill 探讨更好的组织方式

**重要**：新topic的创建必须经过用户明确同意。

## 阶段 6: 准备写作素材

### 6.1 收集所有素材

整合以下内容作为写wiki的素材：
1. 对话中提取的核心概念和解答
2. 阶段3评估中确认的正确理解
3. 阶段4中纠正后的理解
4. 用户希望重点记录的要点

### 6.2 规划页面结构

根据素材内容，规划wiki页面的结构：

**常见页面结构模板：**

**概念解释型：**
```
# [概念名称]

## 核心机制
[概念的核心原理]

## 常见应用
[使用场景和示例]

## 常见误区
[容易误解的点，基于阶段3-4的评估]

## 与其他概念的关系
[相关概念的wikilinks]
```

**Q&A型（适合对话内容较松散）：**
```
# [主题名称]

## 核心问题
[Q1: 问题 → 解答]
[Q2: 问题 → 解答]
...

## 关键要点
[从对话中提取的核心要点]

## 常见误区
[基于评估发现的误解]
```

### 6.3 与用户确认写作要点

向用户展示计划写入的内容要点：

```
📝 写作计划

页面位置: [topic]/[页面名]
页面类型: [概念解释型/Q&A型/其他]
主要内容：
  - [要点1]
  - [要点2]
  - [要点3]
  
是否需要调整？
```

## 阶段 7: 执行写作

### 7.1 创建/更新Wiki页面

直接执行写作任务：

**执行步骤：**
1. **调用 wiki-voice skill**：确保语言风格符合要求
2. **撰写页面内容**：使用中文（技术术语保持英文）
3. **添加双向链接**：
   - 链接到相关现有页面 `[[相关页面]]`
   - 在相关页面中添加指向新页面的反向链接
4. **更新索引**：
   - 更新 `wiki/[topic]/_index.md`，添加新页面条目和一句话摘要
   - 更新 `wiki/_master_index.md`（如果创建了新topic）
5. **更新日志**：
   - 向 `wiki/[topic]/_log.md` 追加记录
   - 向 `wiki/_log.md` 追加记录

**日志格式：**
```
## [YYYY-MM-DD] COMPILE | LLM对话学习: [概念] | 创建/更新 [页面名]
```

### 7.2 验证输出

执行完成后，检查：
- 页面是否创建成功
- `_index.md` 是否更新
- `_log.md` 是否追加记录
- wikilinks 是否正确
- 内容是否符合 wiki-voice 风格

## 阶段 8: 总结

向用户汇报：

```
✅ 学习整理完成

📄 创建/更新的页面:
   - [页面路径]

📊 本次学习概况:
   - 核心概念: [概念]
   - 掌握度: [评估结果]
   - 纠正的误解: [数量及简述]

🔗 相关链接:
   - [[相关页面1]]
   - [[相关页面2]]

💡 后续建议:
   - [如果有]
```

## 特殊情况处理

### 对话内容质量差

如果对话内容过于零散、缺乏实质内容，或主要是闲聊：
- 向用户说明："这段对话似乎缺少可以系统化的知识内容"
- 询问："你是否想补充一些要点，或者这只是初步探讨？"

### 用户已经熟练掌握

如果阶段3评估显示用户已经完全掌握：
- 简化评估流程
- 重点放在"记录精要"而非"纠正理解"
- 可以建议："看起来你已经很熟悉了，要快速记录一个要点笔记吗？"

### 多概念混合

如果一次对话涉及多个不相关的概念：
- 分别评估每个概念的掌握度
- 建议分多个wiki页面记录
- 为每个概念单独执行阶段5-7

## 与现有技能的协作

| 阶段 | 调用的技能 | 用途 |
|------|-----------|------|
| 1.2 | agent-browser | 获取完整对话内容 |
| 5.3 | brainstorming | 探讨wiki组织方式（可选） |
| 7.1 | wiki-voice | 保持语言风格 |
| 全程 | - | 直接执行，无需计划工具 |

## 语言要求

- **本技能的所有输出（除wiki内容外）**：可以使用中文或用户偏好的语言
- **Wiki页面内容**：必须用中文撰写（遵循 AGENTS.md 的 Wiki Content Language 规则）
- **技术术语**：保持英文，首次出现时简要解释
