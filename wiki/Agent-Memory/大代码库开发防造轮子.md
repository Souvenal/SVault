# 大代码库开发防造轮子

*AI Agent 在大型代码库和二次开发场景中避免"造轮子"的策略与规则。*

---

## 问题场景

在以下场景中，AI Agent 极易产生"造轮子"行为：

- **UE5 游戏开发**：C++ 代码库极其庞大，已有丰富的函数、类、结构可复用
- **游戏引擎开发**：模块众多，层次复杂
- **基于 LLVM 开发**：基础设施本身就极其庞大
- **二次开发**：在他人项目中继续开发，原作者可能已做了大量封装

**典型模式**：Agent 在没有充分 grasp 已有基础设施的情况下，基于幻觉或已有经验开始规划，产出方案中大量新增方法、结构、类，而非复用现有代码。经人工 review 后指出已有接口，Agent 才能给出更合理的方案。

**本质**：这不是"造轮子"的道德问题，而是 **Agent 缺乏充分的上下文理解就急于编码** 的工作流缺陷。

---

## 解决方案

### 方案一：工作流强制"先探索再编码"

这是最根本的解决方案。Claude Code 官方最佳实践明确倡导的 **"Explore → Plan → Implement"** 三阶段工作流：

#### Claude Code 的 Plan Mode

Claude Code 内置的 Plan Mode 是天然的"防造轮子"机制：

1. **Explore 阶段**：在 Plan Mode 下，Agent 只读取文件、理解结构，**不进行任何更改**
2. **Plan 阶段**：基于探索结果制定方案，明确指出哪些现有模块可复用、哪些需修改
3. **Implement 阶段**：切换回 Normal Mode 执行

#### Sabrina Ramonov 的 qplan 工作流

来自 Sabrina Ramonov 的实战方法论（[[https://www.sabrina.dev/p/ultimate-ai-coding-guide-claude-code|原文]]），核心是三个快捷命令：

**`qnew`**（会话初始化）：
> Understand all BEST PRACTICES listed in CLAUDE.md. Your code SHOULD ALWAYS follow these best practices.

**`qplan`**（计划验证，**防造轮子的核心**）：
> Analyze similar parts of the codebase and determine whether your plan:
> - is consistent with rest of codebase
> - introduces minimal changes
> - **reuses existing code**

三个评估维度直接对应防造轮子：一致性、最小变更、代码复用。

**`qcode`**（实施）：
> Implement your plan and make sure your new tests pass. Always run tests to make sure you didn't break anything else.

#### Subagent 探索

Claude Code 支持在独立上下文窗口中启动 Subagent 进行深度探索，不污染主对话上下文：

```
Use subagents to investigate how our authentication system handles 
token refresh, and whether we have any existing OAuth utilities I should reuse.
```

### 方案二：利用上下文引擎增强代码库理解

上下文不足是造轮子的根本原因。部分工具专门解决了大代码库的上下文检索问题：

#### Augment Code 的上下文引擎

Augment Code 的核心理念是 **"上下文优于智力"**：

- **全量摄取 + 语义嵌入**：不仅做关键词匹配，而是理解代码间的逻辑关系
- **毫秒级同步**：代码变更实时反映到上下文中，避免基于陈旧代码提建议
- **提示增强**：自动将简单用户提示丰富为包含代码库上下文的详细查询
- **效果**：新工程师 onboarding 时间从 4-5 个月缩短至 6 周

#### Sourcegraph Cody

基于 Sourcegraph 代码搜索引擎的 AI 助手，核心优势在大规模代码库的**跨仓库上下文检索**，比 GitHub Copilot 更擅长"Codebase Understanding"。

### 方案三：规则驱动——将"防造轮子"写入用户级 Memory

这是最实操的方案：将防造轮子的策略写为规则，写入各工具的**用户级配置**（参见 [[用户级配置指南]]），确保每次对话自动注入。

#### 规则模板

以下是一套可直接写入用户级配置的规则：

```markdown
# 代码复用与防造轮子规则

## 核心原则：先探索，再编码

1. **禁止盲目新增实体**：在新增任何类、函数、结构体之前，必须先搜索代码库中是否已有可复用的实现
2. **最小变更原则**：优先修改现有代码而非新增代码；优先复用已有接口而非创建新接口
3. **模式一致性**：新代码必须与代码库中已有的模式和风格保持一致

## 执行流程

### 大代码库开发（>1000 行或使用大型框架如 UE5/LLVM）
- 必须先进入 Plan Mode 或探索模式，理解项目架构和已有封装
- 在规划方案中，每个"新增"决定必须说明为什么不能复用已有实现
- 方案中必须包含"已有可复用接口"清单

### 二次开发场景
- 必须先阅读项目的主要封装层和公共 API
- 在规划中明确标注：哪些是已有接口调用，哪些是新写代码
- 新写代码比例超过 30% 时，必须重新审视是否有可复用方案

## 具体操作

- 新增函数前：搜索 `grep -r "类似功能关键词" --include="*.cpp" --include="*.h"`
- 新增类前：确认代码库中没有功能相近的类可以扩展或组合
- 新增工具函数前：检查项目是否已有类似的 utility/helper 库
- 使用第三方库前：确认项目依赖中是否已包含该功能

## 违规信号（需要人工 review）

- 方案中出现大量新文件/新类
- 新增的函数功能与已有函数高度重叠
- 未引用项目已有的基础设施工具/封装层
```

#### 各工具配置方法

| 工具 | 配置位置 | 方式 |
|:---|:---|:---|
| Claude Code | `~/.claude/rules/code-reuse.md` | 模块化规则，`alwaysApply: true` |
| CodeBuddy | `~/.codebuddy/rules/code-reuse.md` | 模块化规则，`alwaysApply: true` |
| Codex | `~/.codex/AGENTS.md` | 追加到全局配置文件 |
| OpenCode | `~/.config/opencode/AGENTS.md` | 追加到全局配置文件 |
| Cursor | Settings → User Rules | 在 GUI 中粘贴 |

对于 Claude Code 和 CodeBuddy，推荐模块化方式（独立规则文件），因为更易维护和跨机器同步。

### 方案四：项目级 AGENTS.md 引导

GitHub 基于 2500+ 仓库的分析（[[https://github.blog/ai-and-ml/github-copilot/how-to-write-a-great-agents-md-lessons-from-over-2500-repositories/|原文]]）给出了编写 AGENTS.md 的最佳实践，其中与防造轮子直接相关的：

1. **提供可执行命令**：让 Agent 直接复用项目的构建、测试、检查流程，而非自己发明
2. **详细说明项目结构与技术栈**：让 Agent 知道代码应放在哪里、遵循哪种风格
3. **展示代码示例**：一个真实的代码风格片段胜过三段文字描述，让 Agent 学习和复用已有模式
4. **设定明确边界**：`🚫 Never do: Create new utility functions without checking existing ones first`

#### 推荐的项目级 AGENTS.md 结构

```markdown
# 项目名称

## 架构概览
[简述项目架构、模块关系、核心封装层]

## 关键复用接口
- `Utils::Logger` — 日志系统（不要自己写 logging）
- `Framework::EventBus` — 事件系统（不要自建观察者模式）
- `Core::ResourceManager` — 资源管理（不要手动管理生命周期）

## 编码规范
[提供真实的代码风格示例]

## 构建与测试
- 构建：`cmake --build build`
- 测试：`ctest --test-dir build`

## 禁止事项
- 🚫 不要创建新的日志工具，使用 `Utils::Logger`
- 🚫 不要创建新的单例模式，使用 `Framework::Singleton`
- 🚫 不要引入未经确认的第三方依赖
```

### 方案五：跨工具规则复用

参见 [[https://yuluyao.com/ai/cursor-claude-codex-reuse-rules|三款 AI 工具复用规则]]，推荐结构：

```
docs/ai/
  code-reuse.md              # 防造轮子规则（共享）
  project-context.md          # 项目背景（共享）
AGENTS.md                     # 跨工具简单入口
CLAUDE.md                     # Claude Code 入口（引用共享文档）
.cursor/rules/
  code-reuse.mdc              # Cursor 规则（引用共享文档）
```

核心思想：**写一份共享规则文档，为每个工具配薄适配层**。AGENTS.md 作为跨工具入口（已获 Linux Foundation 下 Agentic AI Foundation 维护，被 60k+ 开源项目使用），但不要堆砌所有规则，而是作为索引引用具体共享文档。

## 效果评估

| 方案 | 侵入性 | 有效性 | 适用场景 |
|:---|:---|:---|:---|
| 工作流强制探索 | 低 | 高 | 所有场景，尤其大型代码库 |
| 上下文引擎 | 中（需换工具） | 最高 | 超大代码库、monorepo |
| 用户级规则 | 低 | 中 | 跨项目个人习惯 |
| 项目级 AGENTS.md | 中 | 中高 | 团队协作项目 |
| 跨工具规则复用 | 中 | 中 | 多工具切换场景 |

**最佳组合**：用户级规则（兜底）+ 工作流强制探索（主动）+ 项目级 AGENTS.md（团队共识）。
